Software-Defined Networking (SDN) とネットワーク機能仮想化 (NFV) のトレンドは完全なフレームワークを形作る様に進化し収束市続けており、将来の風景はより鮮明になっている。 SDN と NFVは — 単一のコントローラで分散 IP ルーティングを代替えするとか、”高度なネットワーク機能を独自の SMP から PC に移植する” といった — 単純なアイデアによって始動されたにも拘わらず、これらの以前の流行語の背後にある推進力は着実に増大している。
これは SDN/NFV 観念が今やずっともっと関係していると理解されているという事実にも拘わらずだ。 ここで行われている事のより深い基本原則の幾つかを見る事によって、私達はこれらテクノロジーの背後にある推進力についてより良い理解を得る事が出来る。 更に <a href=”https://www.sdncentral.com/listings/google/” target=”_blank”>Google</a> や <a href=”http://www.apple.com/” target=”_blank”>Apple</a> のようなテクノ巨人が過去5年の間に数百万台の携帯電話に数十万の機能をどのようにして付加する事が出来たかを見る事によって、私達は同様の原則が SDN と NFV のプログラマビリティ及び拡張性を増大させる為にどのように働くかについて幾らかの洞察を得る事が出来る。
大きくは <a href=”http://www.apple.com/ios/” target=”_blank”>iOS</a> と <a href=”http://www.android.com/” target=”_blank”>Android</a> がどのようにして前述の記念碑的な携帯電話の仕事を成功させたかは彼らがモバイルのコンテキスト管理を再定義した方法に大いに関係している。 コンピュータの中のプログラムと違って、現在コンテキストスイッチをフロント使用しているスマートフォンオペレーティングシステム (OS) 上のアプリは他のアプリ用のバックエンドプロセス(デーモン)によって妨害される事はない。 この分離によって膨大な量の開発者や独立系ソフトウェアベンダ (ISV) は足並みを揃えたリグレッションテストをジョイントで行う必要性なしに、そして互いのユニークなユーザエクスペリエンスに影響を与える事無くアプリケーションを発売出来る。 — 開発者が機能をルーティング機器から仮想マシンへアンバンドルする時に欲しい種類の保護インフラストラクチャだ。
モバイルアプリにおいては、かつて干渉と悩ましい程の応答の遅さを生じた古い重いバックグラウンドプロセスロードは今やクラウドに再配置されており、そこでは、ユーザのアプリ/電話接続のコンテキストに基づいて、バックエンドプロセスは Angry Bird のハイスコアからカレンダの同期、音楽の蓄積に至るまでのあらゆる物を管理している。 これら全てを共有し、非フラグメントでありながら完全に分離された方法で管理する能力はネットワークの高機能プロセスをルーティング機器からコンピュータクラスタの中へアンバンドルする為のもう一つ高く評価されるべき品質だ。
<h2>コンテキストの同心円状リング</h2>
クラウドプロバイダとキャリアネットワークのインフラストラクチャを良く観察する事によってステートとコンテキストの管理についての様々な方法に気付く。 丁度モバイルの消費者の場合の様に、これらのアプローチは発展しており、SDN/NFV のスコープの中で再定義されているので、より良いアジリティ/パフォーマンスと独立系ソフトウェアベンダによるコンカレントな開発が可能である。
例えば、<a href=”https://www.sdncentral.com/listings/facebook” target=”_blank”>Facebook</a> のようなインターネット巨人のデータセンタにおいてはユーザアプリケーションコンテキストの管理専用の非常に沢山のコンピュータラックを見る事ができる。: 誰がどのインデックスを参照しているか、又は誰がどの友人のウォールに何をポストしていたかを管理している。 ところが、典型的なコアルータの中では最低限の “コンテキストフリー” なデザインを見る。 このデザインの目標は “パケットについての全てから開放される” という言葉に要約される。 私達は今やこれらの伝統的な両極端が SDN によって挑戦され再定義される地点に到達し、コンテキスト管理セグメンテーションはスマートフォンの導入のケースに多く似ている。
これらの進化したコンテキスト(再)セグメンテーションを処理する為に出現したアーキテクチャは同心円状のリングに似ており、そこではセグメンテーションは主にエンド・ツー・エンドアプリケーションステートのどの部分が保持されているかによって定義されている。 例えば、最も内側のコンテキストサークルは依然高密度の IP コアスイッチやコアルータだ。 この下部基盤は最小のローカライズされた “知る必要がある” ステートを持っており、小さく高密度のボックスに転送する為に数テラバイトを圧縮する事が出来る。他方、最も外側のリングでは操作-アプリケーション当たりのユーザコンテキストをマップリデュースする為の専用の大量のデータセンタラックを必要とするクラウドアプリケーションがある。 中間では、SDN/NFV オーバレイを定義してコンテキスト管理の視点からそれらがどのようなものかを調査する。
NFV オーバレイはキャリアと他のサービスプロバイダが固定/モバイル顧客とインターネットアプリケーションの間に中間機能を挿入する方法についてが全てだ。 これらのユーザ接続はパブリックネットワークのロードを増大させるので、その使用量を調節しポリシーを適用する沢山の機能がある。 全ての仮想ネットワーク機能が同じ様に作られているわけではないのは明らかだ! IP テレフォニーの様な仮想ネットワーク機能 (VNF) は本格的なピア・ツー・ピア通話アプリケーションを形成し、中間機器以上のものだ。 更に、通話相手が居なくてボイスメールに振り向けられた場合には、これは完全なストレージ、インデクシング、符号変換アプリケーションとなるだろう。 他の機能としては軽いタッチのヘッダー調整による重要な加入者モビリティの提供から、非常にコンピュータに近い集中的なフルキャッシュの Web プロキシー迄の中間機器機能がある。
ともかく、これらの機能はコアキャリアネットワークとインターネットプロバイダの間の範囲のアクションを実行するのに十分なコンテキストを維持している。 これらの機能は今や高度に保護された仮想マシンを使用して提供出来るので、共有される標準のハードウェアの上で最小の干渉で実行出来る。 これはネットワークインフラストラクチャを “その為のアプリ” モードと力学に転換し、同時の開発を保証する。
機能処理ユニット (NF-VM) によって保持されたコンテキストの量がユニットが受け取りそして再転送するであろうトラフィックの量を決定する。 これは又、最終的に下部基盤となるソフトウェア・ネットワークからの NFV 要件を定義する。
<h2>コンテキストが SDN/NFV リンクを定義する</h2>
この点をより明確にする為に、エボルブドパケットコア (EPC) を例に取ってみよう。 EPC はインターネットがそう “想定していない” 一方、ユーザが移動出来る事に専念したモバイルブロードバンドとインターネットの間の(仮想化)機能の集合だ。 これはインターネットへの接続に対して、ネットワーク内のユーザコンテキストを(パケットのハブとして)ローミングユーザ毎に維持する事によってなされる。 これに伴うモビリティ管理機能はユーザ管理のコンテキストを維持するけれども実際のデータパスパケットをチャネルしない。 明らかに各ギガビット/秒のトラフィック当たり保持されたユーザコンテキストの量は二つのサブ機能の間で大きく異なる。 それらは又、ユーザのタイプ、例えばメーターかスマートフォンかによっても異なる。 VM 毎のコンテキストアフィニティが下部基盤となる Software-Defined Network を NFV ソリューションとリンクする為の方法と要件を決定する。
基盤となるソフトウェアネットワークは適切なトラフィックを適切な順序で送る為に、明らかに VM 毎の NFV のタイプとユーザコンテキストをまさに十分に知る必要がある。 コアスイッチは明らかに通常のブリッジングとルーティングスタックに基づいて動作するのではなく、その為に結線されてはいないけれど、これらのスタックは基本的な接続性を実装する為に依然、非常に必要とされる。 従ってこれらのスタックの上でネットワークから必要とされるものはコンテキストアウェアアプリケーションネットワーキングの為の専用のオーバレイだ。 これは私達がキャリア SDN を定義しているものそのものであり、又、基本的な IP パケット接続上で機能処理をするアクセスからインターネットへのフローフォワーディングの定義だ。
最後に、包括的に眺める為に、私達はコンテキストのスケーリングについて検討しなければならない。 最内側のコアネットワーキングの下部基盤はある程度自身でスケールしなければならず、結果的に管理するステートとコンテキストの量を最小化するだろう。 これは、もしゼロから自分で行なった場合には特に困難なタスクになり得るからだ。 SDN オーバレイはそのようなオプションを持っていない。 SDN は、定義によって、かなりの量の運ぶべきコンテキストを持っている。 然しながら、下部基盤となるネットワークによってスケールされる軽量のコンテキストマッピングシステムを構築する事によってアンダーレイとなる IP レイヤを補助として使用できる。 これは <a href=”https://www.sdncentral.com/listings/opendaylight-project” target=”_blank”>OpenDaylight</a> がキャリア SDN のフェデレーションをスケールするのに使用している Location/ID Separation Protocol (LISP) オーバレイ標準のコアアーキテクチャだ。 Although the NFV アウターレイは SDN よりもかなり多くの運ぶべき長命のステートとアプリケーションコンテキストを持っているけれども、丁度データセンタアプリケーションコンテキストが下部基盤となるマップリデュース用インフラストラクチャによってスケールされるように、それは実際には SDN レイヤによってスケールされる。
接続機器が普及し、スローダウンする兆しがない今日、今や SDN と NFV テクノロジーがそれにならい、サービスプロバイダインフラストラクチャを次のレベルに進める時だ。
