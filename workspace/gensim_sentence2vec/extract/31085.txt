3.1.x か 3.2.x ソリューション   
問題を解決する 3 つの方法があります。 これらのソリューションは次の セクションで説明されます。 さらに、賛否両論は各ソリューションのためにリストされています。 
ソリューション 1 — Cluster-A および Cluster-B の疑似ゲートキーパーの作成は、それから「番号非通知呼び出し」を可能にします   
最初のソリューションは匿名デバイスを可能にするゲートキーパーコンフィギュレーションを含みます。  
このソリューションはクラスタ内部 エンドポイントがのそれぞれを H.323 未知エンドポイントから着信コールを受信するようにします。 従って、CM-A2 からのコールは受け入れられ、確立することができます。  
ゲートキーパーを設定するとき、管理 ガイドによって必要なフィールドに入力して下さい。  
注: このスクリーン ショットでマークされる割り当て番号非通知呼び出しおよびデバイス・ プロトコル フィールドに特別な注意を注意して下さい、:  
賛成論 
簡単なコンフィギュレーション—コールルーティング設定の修正が必要となりません。 
スケーラビリティ—単一 の デバイス 設定。 
スケーラビリティ—ただネットワークにそれを接続するために新しいクラスタを設定する必要があります。 
反対論 
冗長性—明示的に 設定されたサーバの失敗の場合の送信 冗長性無し、(この例、CM-A1 または CM-B1 で)。 リモート クラスタへの単一経路。 
優雅— 「擬似」ゲートキーパーが作成されるので、ないソリューションを「きれいにして下さい」。 
注意: 未知インタークラスタ トランク エンドポイント(例を許可します: クラスタに呼び出しを確立する未知 Cisco CallManager クラスタ)。  
ソリューション 2 —各々のリモート Cisco Unified CallManager のためのインタークラスタ トランクの明確な設定   
このソリューションはそれぞれがすべてのリモート Cisco CallManager サーバを指す各 Cisco CallManager クラスタの明示的な クラスタ間トランクを設定するように要求します。 呼ルーティングを修正することは必要ではないです; それは単一 インタークラスタ トランクと同様に変わらないはずです。  
このソリューションは各クラスタがリモートサーバ「に」気づくようにします。 従って、CM-A2 からのコールは受け入れられ、確立されます。 賛成論 
簡単なコンフィギュレーション—コールルーティング設定の修正が必要となりません。 
反対論 
冗長性—プライマリ・ サーバの失敗の場合の送信 冗長性無し、(この例、CM-A1 または CM-B1 で)。 
スケーラビリティ—準完全 メッシュが必要となります。 
スケーラビリティ—新しいクラスタがネットワークに追加されるときすべてのクラスタを再構成して下さい。 
ソリューション 3 —ルート・ リストに含まれている各々のリモート Cisco Unified CallManager のためのインタークラスタ トランクの明確な設定   
このソリューションはそれぞれがすべてのリモート Cisco CallManager サーバを指す各 Cisco CallManager クラスタの明示的な クラスタ間トランクを設定するように要求します。 呼ルーティングを修正することは必要です: Route リストを作成して下さいプリファレンスの順ですべての設定されたインタークラスタ トランクが含まれているルート グループが含まれている。 この Route リストはクラスター間 コール・ルーティングに責任があるルートパターンの宛先であるはずです。 
このソリューションは各クラスタがリモートサーバ「に」気づくようにします。 従って、CM-A2 からのコールは受け入れられ、確立されます。 それはまたプライマリ トランクかサーバがダウン状態になれば呼び出しがバックアップ トランクで確立されるようにします。 賛成論 
冗長性—プライマリ・ サーバの失敗の場合の送信 冗長性、(この例、CM-A1 または CM-B1 で)。 
反対論 
複雑な状況—コールルーティング設定の修正が必要となります。 
スケーラビリティ—準完全 メッシュが必要となります。 
スケーラビリティ—新しいクラスタがネットワークに追加されるときすべてのクラスタを再構成して下さい。 
3.3.x 機能拡張および設定   
インタークラスタ トランク プロトコルを使用する必要があるとき Cisco Unified CallManager 3.3.x は冗長 な インタークラスタ トランクを設定することもっと簡単にする新しい設定 インターフェイスをもたらしました。 以前の CallManagerのバージョンでは同レベルの冗長性を追加するために、複数のゲートウェイがおよび複雑なコンフィギュレーションは、必要となりました。 
複数 の IP エンドポイントの単一 インタークラスタ トランクの設定   
Cisco Unified CallManager 3.3.x インタークラスタ トランク 設定は管理者が 3 つまでのリモート CallManager Server の IP アドレスで各クラスタの単一 の ゲートウェイを作成するように要求します。 この単一 の ゲートウェイはそれぞれクラスタですべての CallManager Server に利用できます。 優先順位を設定 するために、好む順序でリモート CallManager を設定して下さい。 
CallManager 3.3x のトラックの設定のその他の情報に関しては、トランク の 設定を参照して下さい。
