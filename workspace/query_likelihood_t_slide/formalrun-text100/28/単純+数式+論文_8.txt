究室によっては,これとは異なる方針を持っておられるところもあると思うので,注意して欲しい.
「スタイルなんてどうでもいいじゃん」などとは,決して思わないで欲
しい.
多くの場合,体裁も整っていない論文は「きっと中身も出来が悪いに違いない」
という先入観を与えがちである.そして,私の経験からすると,
その先入観はしばしば当たっている.
もちろん,整っているからといってそれがいい論文であるとは言えないが,体裁
に問題があるようでは,すでに「読みやすさ」の点で論文の備えるべき資質を欠
いている.

おまけ:
論文の提出までの作法
1.タイトル
論文のタイトルは,あまり範囲を広げ過ぎず,
自分のやった仕事を的確に表すものを考えることが重要である.
ただし,あまり長すぎるのもよろしくないので,なかなか難しい.
タイトルのスタイル
Studies on … (〜に関する研究)
A Study on …
あるテーマについて,高い観点から総合的な研究を行ったときに用いる.
ただ,「A Study …」は,あまり広範囲に研究しなかったような印象を 与えかねないので,天野はあまりお薦めしていない.
Design and Implementation of …(〜の設計と実装)
何かのアイディアに基づいて新しいシステムを開発したときなどに用いる.
もっと具体的にアイディアの内容を述べるものもある
(例)Dynamic Performance Optimization for … (〜のための動的性能最適化)
1は学位論文系によく見られる形式.
2や3は論文誌・国際会議によく見られる形式
(学位論文でも十分に内容があればもちろんOK). 
タイトルのように短い表現の中に同じ単語が2回以上出るようなら,一度,タイトルを考え直してみるとよい.
もちろん,絶対にダメだと言っているわけではなく,あくまで,ケースバイケースである.
タイトルの長さに具体的な制限はないが,日本中の研究者が毎年提出している科学研究費の計画調書(申請書)に記載できる研究課題名の長さが
全角 40
字までとされていることを考えると,これに収まるようにしておく,というのは,ひとつの目安になるだろう.
表紙において,センタリング(中央揃え)されたタイトルが単語の途中で切れる
(自動改行される)ようなときには,適切なところで強制改行することによって,見苦しくならないように整形する. 
2.論文の構成
第2章では,卒業論文・修士論文の標準的な章立てについて述べる.
(第X章のタイトルと第X.1節の間の部分には,
こんなふうにその章の概要説明を書くようにするとよい.
また,わざわざ節立てする程もなさそうなこと(説明)もあれば,
ここに書くとよい。)
2.1   章の構成
章タイトルと第1節の間
その章の概要説明や,わざわざ節立てする程もなさそうなことを書く。 
各章の構造
第N章が長くなる(何ページにもなる)ときは,適切な表題をつけて,節に分割する. 
N.1 
N.2 (LaTeXで書くときは,これらは\section{}になるだろう) 
N.3 
などのようになる.第N.M節は,さらに 
N.M.1 
N.M.2 (LaTeXで書くときは,これらは\subsection{}になるだろう) 
N.M.3 
などと分割してもよい.ただし,よほどの大長編でないかぎり,これをさらに分割して
N.M.K.1 などを作るのは避けたほうがよい. 
2.2   論文の標準的な構成
まず,冒頭には,章番号のつかないパートがある.
表紙(タイトル,氏名,所属)
ページ番号なし. 
概要(Abstract)-----1ページ以内
ここからローマ数字のページ番号が付く. 
目次
本文に登場する概要,章,節,謝辞,参考文献,付録などのタイトルとページを
掲載する.「図目次」と「表目次」は,入れなくてよいと思う. 
ここまでの概要と目次には章番号を付けない.また,通常,概要と目次までのペー
ジ番号には,ローマ数字(i, ii, iii, iv, ...)を用いる.
これ以降,章番号を付け始める.また,ページ番号はアラビア数字(1, 2, 3,
...)に切り替わり,リセットされて1から始まる.
まえがき(はじめに、序論)(Introduction)
-----最低でも2〜3ページ
研究の背景、動機 
研究の目的、目標 
この研究で採用する方法、手法の簡単な説明 
得られた結果の概要 
論文の構成 
(基本的事項)
-----(論文の大半をこの章が占めるようなことにならない限り)好きなだけ書く。
----- 5ページでは少ない。
「基本的事項(Basic Concepts)」そのものをタイトルにするのは,理論系の論文
ではときどき見かける.
その場合,ここでは,記号の意味や数学的・論理的な概念の説明が(他の
参考文献を読まなくてもよい程度に)詳細に説明される.一方,システム系の論
文で「基本的事項」というタイトルを使うのはあまりポピュラーでないので,もっ
と内容を的確に表すタイトルを付けること.「,および××」というよ
うな三題噺のようなのはできれば避けたいが. 
従来の手法を解説(言語、システム、etc...) 
この研究で採用した言語やシステムの説明(構文、機能、制約、 etc...) 
研究内容の具体的説明
-----好きなだけ書く.ただし,20ページにもなるようなら,章の分割を考えること.
実験で性能評価をするときには,その説明は次の章(4章)で別にしても よい.
むすび(おわりに、結論)(Conclusion,Concluding Remarks)
-----最低でも1〜2ページ
この章の表題はそれぞれ1章に対応した語句で。 
まえがき ⇔ むすび
序論 ⇔ 結論
はじめに ⇔ おわりに
研究の目的・目標と,それをどのように,どこまで達成したか,まとめる.
さらに,この成果の応用・効果としてどのようなことが考えられるか,述べる.
最後に今後の課題を述べるのもよいが,あまり適当なことを書かないこと.何が未解決で,
それを解決するために(場合によっては自分の後を引き継ぐ人が)どのようなことに取り組むべきだと自分が考えているか,きっちり書くこと. 
4章を「実験および性能評価」などにしたときは,ここは5章になる.もう少し章が増えることもあるかも知れないが,
章を分割しすぎて各章があまり散慢にならないように. 
これ以降の章には,章番号を付けない(LaTeXでは,\chapter*{}になるだろう).
謝辞(Acknowledgment)
いろいろなしがらみから,感謝の意を述べる順序には細やかな配慮が必要である.
研究室の先輩,そして,実質的に指導してくれた先生と正式の指導教員が異なる
場合にはその実質的に指導してくれた先生に,相談することをお勧めする. 
参考文献(References, Bibliography)
最低でもB4は10件、M2は20件.
LaTeX で thebibliography環境を使うなら, \chapter*{}コマンドは必要な
い. 
付録(Appendix)-----あれば
自分のやった研究の成果で,
論文本体に長々と書くほどではないようなものを書く。
例:プログラムのコード全体,本文中には掲載しなかった出力例の図(数が多い
とき)など. 
ただし,プログラムのソースコードなどを入れる場合には,タイプライタフォン
トを用いるか,そのプログラミング言語のソースプログラムの表示に適したスタ
イルファイルを用いて見やすく整形する.通常のテキストと同じ形式では,はな
はだ読みにくい. 
また,ソースコードの文字サイズは,論文本文の文字サイズよりも十分小さくす
る.これは「こんなのでページ数をかせぎやがって…」と思わせないための自衛
策. 
索引(Index)-----あれば
ぜひ頑張って書いてもらいたい.LaTeX の makeindex (mendex) は便利.
ただし,重要なキーワードを上手く選ぶこと. 
3.本文執筆上の注意
よく「何ページあったらいいですか?」など
と聞いてくる学生さんがいる.こんなこと,聞いてくるほうが間違っている.論
文の質が重要なのに,ページあれば OK
なんて基準があるはずないではないか!
しかし,以下のようなことも言えるだろう. 
くり返しになるが,ページ数に関し,以上なら OK などという基準は存
在しない.極端な話,わかりやすくて,それでいて,試問に立ち会う先生 たちを「あっ」と言わせられる内容のある論文だったなら,20ページだっ
たって構わない.しかし,先生たちも驚くような内容がたった20ページで 語りつくされることは,まずあり得ない.そんなのは,きっと説明不足
で難解な論文になっているだろう. 
半年,1年,2年,論文にかけた時間は人それぞれだろうが,その成果が
たったページに収まってしまうなんて,あなたはそれで哀しくないか?  恥ずかしいと思わないだろうか? ページ数を自分で決めるときには,ぜ
ひそのような観点で考えて欲しいと,切に願う. 
もちろん,量が多ければいい,ということも決してない.特に,ページ数を
増やすために涙ぐましい努力(ズル)をしている論文を見ると,悲しくなってしまう.だが,残念なことに,先生たちもプロである.学生さんたちが
ちょっと工夫したくらいでだませる相手だなどと甘く見ていると,本当に痛い目に会いますゾ. 
日本語論文の場合,論文中で使用する句読点は統一する. 
「,」と「.」(カンマとピリオド)
文中に英単語,変数名が現れるときは,こちらのほうが美しいと思う.
「、」と「。」(てんとまる)
日本人なら絶対こっちだ!という人も中にはいるだろう.
「,」と「。」(カンマとまる)
(文部科学省関係の公文書にはこれもある.が,私はお勧めしない.)
とにかく,大事なのは,上のどれかのルールを論文全体で統一的に使うこ と. 
図や表には,必ずキャプションを付ける. 図のキャプションは図の下, 表のキャプションは表の上
というのが常識的な配置である. 
図や表を入れたら, それらに必ず本文中で言及して,図表に対する説明を加えるか,図表を用い
た説
明をすること.
本文で言及されないような図表は,本来,その論文には必要ない
(=なくても困らない)はずなので,そんなものは削除してしまうべきである.
また,当り前のことだが,実際に図表に付けられているキャプ
ションと,本文中での言及は一致していること.番号がずれたり違ったりしてい
ては,言及されても読み手にはわかりようがない. 
4.参考文献リストの作り方
タイトルとして References と Bibliography のどちらを使うか.
(英語論文の場合)
Referencesの項目は、本文中で必ず参照(refer)している箇所が なければならない. 
Bibliographyではその必要はない。が,その論文を執筆するのに影響を受け
た(可能性のある)すべての文献を列挙することになるので, その分項目はかなり多い方がよい.
本文中での参照のしかた
[1], [2], …… のように通し番号を引用箇所に入れるスタイルと,
[Amano98], [Kaneko03] などのように「第一著者の姓+発表年下2桁」のような 記号を引用箇所に入れるスタイルがある. 
いずれの場合も,本文中に使用した参考文献番号(記号)と,論文末尾の参考
文献リストの各論文の先頭の番号(記号)とが一致していなければならない. 
参考文献番号(記号)が,文の終わりや節の終わりに来るとき(句読点のそば に来るとき)は,句読点の直前に書くのが普通. 
Message passing system transgers data using buffer to store
messages[10]. 
("... messages.[10]" とはしない.) 
MPI の最新版は並列入出力もサポートしているが[7],
必ずしも使いやすいものではない.
(「...が,[7] 」とはしない.) 
参考文献リスト内での並べ方
第一著者名(名字)のアルファベット順(お薦め) 
著者名を Amano, H., Kimura, K.,…
(姓が先,イニシャルが後)のように書くと, 並べ換えもしやすい.
出現順: 執筆中のメンテナンスが大変なのでお薦めしない.
アルファベット順+五十音順(日本語論文の場合のお薦め)
英語と日本語の論文を混ぜこぜにして並べたりしない.
並べ方は統一的に。
著者名の後に半角のコロンを入れたら,その後にスペースも入れる. 
各文献の記述スタイルは統一する。
リスト全体で書き方を統一する.  特に,英語文献の著者名の記述には注意すること. 
given name (first name)を略さずに書く(Hirofumi)か,イニシャル(H.)にするか.
family name (last name)を前に置く(Amano, H.)か,後ろに置く(H. Amano)か. 
刊行物の中に収録された論文1篇を参照する場合
(例) Kimura, K., Amano, H.: "論文タイトル", 書籍名,
Vol. 1,  No. 2, pp. 12−32, July 1999. 
単行本の場合は,なるべく出版社の名前も入れる.
本の名前はできればイタリック体で.
"pp." は "pages" のことなので,何ページから何ページまでなのかを書く. 1ページしかないときは,"p." とする.
論文誌などは,月刊のものが多いので,出版年月まで書く. 
本1冊をまるまる参照する場合
(例) Ullman, J. D.: "Principles of Database
Systems",  Second Edition, Computer Science Press, 1982. 
本の名前はできればイタリック体で.
出版社名は必ず入れる.
月刊でない単行本は,出版年だけでもよいだろう. 
著者名の表記法(姓が先か,イニシャルが先か),出典の書き方,年月の場所など,
すべての文献について統一する.  
空白の制御のしかた(TeXの場合)
(例1) H.~Amano,_K.~Kimura,…
(例2) Amano,~H.,_Kimura,~K.,…
(例3) Vol.~2,_No.~3,_July~1999
~ : ノンブレークスペース
改行して欲しくないところ, 変に広がって欲しくないところに使う. 
_ :
普通のスペース(ふつうは見えないのだが,ここではあえて強調してみた) 
論文の提出までの作法
あまり学生さんに礼儀作法をとやかく言える人間だと自分では思っていないが,
それでも,論文の提出前にできるだけ目を通してよりよい方向に案内してあげる
ために,これだけは守って欲しいと考えていることがいくつかある.
提出締め切り直前になってからどんと全部持って来ない
で欲しい.先生から「論文どうなってる?」と聞かれるまでじっと待ってい ないで欲しい.
先生だって人間なのだ.機械のように高速に読み込んでいっきに朱入れでき
るわけではない.出張や会議や他の学生さんの論文のチェックだってある.自分が修正する時間が確保できるように締め切りよりも余裕を持って返して
欲しい,と思うのなら,その分早めに持ってきてもらいたいし,すぐにレスポンスが欲しいのなら,章ごと・節ごとなど,短時間で返せる量に区切っ
て持ってきてもらいたい.
もし今すぐには持って行けない/メールで送れないけれど,だいたいいつぐ
らいに時間を取って欲しいと思うのなら,事前に相談して欲しい.何度も言 うようだが,残念ながら他にもやらなければならないことはたくさんあって,
突然言われても時間を取れないことは多いのだ.
また,これは本当に申しわけないことなのだが,他の用事で忙殺
されていて,本当にこちらからの催促が遅くなってしまうこともある.教官が 「どうなってる?」と聞いてこないからといって,まだまだ大丈夫だとは思
わないで欲しい.
もし仮に,「あんまり早く持っていくと,大幅な書き直しを求められて大変
だから,ぎりぎりに持っていこう」なんて考えているのなら,それは甘い考 えだ.
もし仮に,こんなのでは卒論・修論として到底許しがたいと考えたら,少し くらい非情だと思われても,時間的に厳しいような修正を要求する覚悟はある.
朱入れされた原稿を受け取るだけでなく,説明や質問を 受ける時間をとって欲しい.
どういう意図でそれを書いたのか,本人に聞いてからでないとコメントでき
ないことがある.「机の上に置いておいていただければ」などと考えず,口頭での説明やアドバイスを受けられるように考えて欲しい. 
修正を求められたら,それがなぜなのかよく考えて,同
じ指摘を繰り返し受けなくて済むようにして欲しい.
これは,主に論文を書く際の文章の書き方とかスタイルに関するコメントに
あてはまることなのだが,同じことを何度も何度も繰り返し修正するのは, 言うほうも聞かされるほうも大変疲れるものだ.また,体裁などよりもその
論文で提案する内容や報告する結果に関する議論をできる限り深めたいと考えている.不毛な時間を減らすためにも,ぜひ協力して欲しい.
ちょっと偉そうに  このようなページ 
を作って公開することにしたのも,そんな気持ちからである.
他人の知的所有権は尊重すること.自分ですべき仕事は 自分ですること.
まず,原著者や著作権所有者の了解を得ない盗作や剽窃は犯罪である.
もちろん,長い論文では第2章,短いものでは第2節で,その論文を理解する
のに必要な事項を整理することは多い.このような場合,他人の著作を要約したり引用したりすることも必要になるだろうが,その場合には,きちんと
参考文献を明示して行うべきである. 
次に,所有者が自由にお使いくださいとばかりに公開している情報ではどう
だろうか? 最近では,webサイトで公開されている情報をファイルに取り 込んでそのまま利用するのも難しくない.時間のないときにはそのような行
為でページ数を増やすという誘惑にかられることもあるだろう.あるいは先輩や友人のファイルからコピーさせてもらう,ということだって可能かも知
れない.
しかし,持ち主の了解があるから何をしてもよいということにはならない.
卒業論文・修士論文の目的のひとつは,提出者がそのような論文を自分でまとめる能力を持っているかどうかを判定することである.みなさんはそれを
自分で証明して見せる義務を負っている.たとえ能力があっても,他人の著作を流用したのでは,それを証明したことにならない.もし流用が露見して
しまったら,自分の能力を証明する機会を自らつぶすことになり,能力を持っていないと推定されても文句は言えなくなるのだということをよく覚えてお
いて欲しい. 
卒業論文・修士論文を書く人へ
